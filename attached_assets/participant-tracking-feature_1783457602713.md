## 1. نظرة عامة على الميزة

### الهدف
تمكين مدير المشروع من تحديد قائمة بالمشاركين المطلوب منهم تعبئة النموذج، ومتابعة حالة كل منهم في الوقت الفعلي، مع إمكانية إرسال تذكيرات مستهدفة للمتأخرين عبر تيليغرام.

### الفجوة الحالية
النظام الحالي يعرف **من عبّأ فعلاً**، لكنه لا يعرف **من كان مفترضاً أن يعبّئ**. هذه الميزة تُغلق هذه الفجوة وتُتيح قياس نسبة الاستجابة الحقيقية.

### أمثلة حالات الاستخدام
- إدارة موارد بشرية تتابع من أكمل نموذج الإجازة من 150 موظفاً
- مدرسة تتابع من أعاد نموذج موافقة ولي الأمر من 200 طالب
- جهة حكومية تتابع إقرارات 80 موظفاً

## 2. المقاربة المختارة: الرابط الشخصي

### لماذا هذه المقاربة؟

| المعيار | الرابط الشخصي ✅ | مطابقة الحقل |
|---|---|---|
| دقة التتبع | 100% — لا التباس | تعتمد على كتابة صحيحة للاسم |
| التكرار | مستحيل — كل رابط يُستخدم مرة | ممكن |
| شخصان بنفس الاسم | لا مشكلة | تعارض |
| تعبئة مسبقة للبيانات | ✅ ممكنة | ❌ |
| إرسال تذكير مستهدف | ✅ بالضبط للمتأخرين | ❌ |

### آلية العمل — خطوة بخطوة

```
1. المدير يُنشئ قائمة المشاركين في لوحة التحكم
          ↓
2. النظام يُولّد token فريد لكل مشارك
          ↓
3. كل مشارك يحصل على رابط شخصي:
   https://domain.com/r/{project-slug}/p/{participant-token}
          ↓
4. الرابط العام للمشروع يُغلق تلقائياً (على مستوى هذا المشروع فقط)
          ↓
5. المشارك يفتح رابطه → النموذج يظهر مع بياناته مُعبّأةً مسبقاً
   (بدون الحاجة لإدخال كود الدعوة)
          ↓
6. بعد التعبئة → يُربط السجل بالمشارك في قاعدة البيانات
          ↓
7. يمكن للمشارك التعديل خلال فترة محددة — بعدها يتحول لوضع القراءة
          ↓
8. لوحة التحكم تُحدَّث فوراً بحالة كل مشارك
```

## 3. تصميم قاعدة البيانات (المحدَّث)

### الجدول الجديد: `project_participants`

```sql
CREATE TABLE project_participants (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id        UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  name              TEXT NOT NULL,            -- الاسم كما أدخله المدير
  identifier        TEXT,                     -- رقم جوال أو إيميل (اختياري)
  identifier_type   TEXT,                     -- 'phone' | 'email' | NULL
                                              -- ✦ مُضاف: يُسهّل منطق الإشعار لاحقاً
  token             UUID NOT NULL UNIQUE DEFAULT gen_random_uuid(),
  telegram_chat_id  TEXT,                     -- ✦ مُضاف: يُربط بعد أن يفتح المشارك رابط البوت
  record_id         UUID REFERENCES project_records(id) ON DELETE SET NULL,
  submitted_at      TIMESTAMPTZ,
  first_opened_at   TIMESTAMPTZ,              -- ✦ مُضاف: أول مرة يفتح الرابط (لم يكمل بعد)
  last_notified_at  TIMESTAMPTZ,
  notify_count      INT NOT NULL DEFAULT 0,
  added_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  notes             TEXT
);

CREATE INDEX ON project_participants(project_id);
CREATE INDEX ON project_participants(token);
CREATE INDEX ON project_participants(record_id);
CREATE INDEX ON project_participants(telegram_chat_id);
```

### التعديلات على جدول `projects`

```sql
-- تفعيل الميزة وإعداداتها
ALTER TABLE projects ADD COLUMN participants_enabled        BOOLEAN  DEFAULT false;
ALTER TABLE projects ADD COLUMN participant_name_field      TEXT;    -- مفتاح الحقل المُعبّأ باسم المشارك

-- ✦ مُضاف: فترة تعديل مستقلة عن edit_token_hours العامة
ALTER TABLE projects ADD COLUMN participant_edit_hours      INTEGER  DEFAULT 48;

-- ✦ مُضاف: رسالة مخصصة تظهر لمن يحاول فتح الرابط العام حين تكون الميزة مُفعَّلة
ALTER TABLE projects ADD COLUMN participant_closed_message  TEXT;
```

### حالات المشارك الثلاث (✦ مُحسَّنة من حالتين)

| الحالة | الشرط | الأيقونة |
|---|---|---|
| **لم يفتح الرابط** | `first_opened_at IS NULL` | 🔴 |
| **فتح ولم يُكمل** | `first_opened_at IS NOT NULL AND submitted_at IS NULL` | 🟡 |
| **عبّأ** | `submitted_at IS NOT NULL` | 🟢 |

> هذا التمييز يُتيح للمدير إرسال رسالتين مختلفتين: رسالة تنبيه لمن لم يفتح، ورسالة تحفيزية لمن فتح وتوقف.

## 4. نقاط API المقترحة (المحدَّثة)

### إدارة المشاركين (للمدير)

| الطريقة | المسار | الوصف |
|---|---|---|
| `GET` | `/api/projects/:id/participants` | قائمة المشاركين مع حالة كل منهم + فلترة بالحالة |
| `POST` | `/api/projects/:id/participants` | إضافة مشارك واحد |
| `PATCH` | `/api/projects/:id/participants/:pid` | ✦ تعديل بيانات مشارك (الاسم / التواصل / الملاحظات) |
| `POST` | `/api/projects/:id/participants/import` | استيراد من Excel مع ربط الأعمدة |
| `DELETE` | `/api/projects/:id/participants/:pid` | حذف مشارك |
| `POST` | `/api/projects/:id/participants/:pid/notify` | إرسال تذكير لمشارك محدد |
| `POST` | `/api/projects/:id/participants/notify-pending` | إرسال تذكير للمتأخرين بشروط ذكية |
| `GET` | `/api/projects/:id/participants/export` | ✦ تصدير تقرير Excel للمشاركين |

### النموذج العام (للمشارك)

| الطريقة | المسار | الوصف |
|---|---|---|
| `GET` | `/api/pform/:projectId/participant/:token` | التحقق من الرابط + تسجيل `first_opened_at` + إعادة بيانات التعبئة المسبقة |
| `POST` | `/api/pform/:projectId/submit` | يُرسل `participantToken` في الـ body — يتجاوز التحقق من كود الدعوة |
| `GET` | `/api/pform/:projectId/participant/:token/view` | ✦ عرض السجل للقراءة فقط بعد انتهاء فترة التعديل |

## 6. الإشعارات عبر تيليغرام

### 6.1 آلية الربط مع المشارك الفرد ✦

تيليغرام الحالي في النظام يُرسل للمدير فقط (قناة/مجموعة). لإرسال رسالة لكل مشارك على حدة:

**الآلية المُوصى بها: رابط `/start` مع الـ token**

```
رابط البوت للمشارك:
t.me/{BotName}?start={participant_token}

الخطوات:
1. المشارك يضغط الرابط → يفتح البوت
2. البوت يستقبل /start مع الـ token
3. النظام يربط chat_id بالمشارك في قاعدة البيانات
4. من الآن — كل تذكير يُرسل مباشرة لهذا المشارك
```

> يُخزَّن `telegram_chat_id` في جدول `project_participants`.
> هذا الربط يحدث مرة واحدة ويبقى طالما المشارك لم يُحذف.

### 6.2 قوالب الرسائل

**أ. رسالة الإشعار المبدئي (توزيع الرابط):**
```
مرحباً {name}،

تمت دعوتك لتعبئة نموذج "{project_name}".
رابطك الشخصي:
{personal_link}

لتفعيل التذكيرات عبر تيليغرام، افتح:
{telegram_link}
```

**ب. رسالة التذكير:**
```
تذكير: {name}،

لم تُكمل تعبئة نموذج "{project_name}" بعد.
رابطك الشخصي:
{personal_link}
```

**ج. رسالة التعبئة الناجحة (تأكيد للمشارك):**
```
✅ شكراً {name}،
تم استلام تعبئتك لنموذج "{project_name}" بنجاح.
يمكنك التعديل حتى: {edit_deadline}
```

## 10. الأسئلة المفتوحة — تحتاج قراراً قبل البدء

> أجب على كل سؤال وستُضاف إجابتك مباشرة للوثيقة لتصبح جاهزة للتنفيذ.

---

**س1 — آلية ربط تيليغرام بالمشارك الفرد:**
كيف يتمكن النظام من إرسال تذكير تيليغرام مباشرة لكل مشارك؟

- **أ)** المشارك يفتح رابط `t.me/BotName?start=TOKEN` مرة واحدة — النظام يحفظ معرّفه تلقائياً *(موصى به)*
- **ب)** المدير ينسخ الرابط ويرسله يدوياً لكل مشارك عبر تيليغرام بنفسه *(أبسط، بدون تعديل في الـ Bot)*

> **الإجابة:** [ ]

---

**س2 — فترة التعديل: مستقلة أم مشتركة؟**
هل فترة تعديل المشارك تُحدَّد بشكل مستقل لكل مشروع، أم تَرث من إعداد `edit_token_hours` الموجود؟

- **أ)** مستقلة — عمود `participant_edit_hours` خاص بالمشاركين *(موصى به — مرونة أكبر)*
- **ب)** مشتركة مع `edit_token_hours` الحالية *(أبسط)*

> **الإجابة:** [ ]

---

**س3 — ماذا يرى المشارك بعد انتهاء فترة التعديل؟**

- **أ)** صفحة عرض بياناته للقراءة فقط — يرى ما أرسله دون تعديل *(موصى به — تجربة محترفة)*
- **ب)** رسالة "انتهت فترة التعديل" فقط

> **الإجابة:** [ ]

---

**س4 — تسجيل لحظة أول فتح للرابط؟**
هل نُسجّل `first_opened_at` لمعرفة من فتح الرابط ولم يُكمل؟ (يُعطي 3 حالات بدلاً من 2)

- **أ)** نعم — 3 حالات: لم يفتح / فتح ولم يكمل / عبّأ *(موصى به — بيانات تشغيلية قيّمة)*
- **ب)** لا — يكفي حالتان: لم يعبّئ / عبّأ

> **الإجابة:** [ ]

---

**س5 — الإشعار المبدئي عند إضافة المشارك:**
كيف يصل الرابط الشخصي للمشارك أول مرة؟

- **أ)** تلقائي — النظام يرسل رسالة تيليغرام فور الإضافة *(يشترط ربط تيليغرام مسبقاً)*
- **ب)** يدوي بضغطة زر — "أرسل للجميع" بعد مراجعة القائمة
- **ج)** المدير ينسخ الروابط ويوزعها بنفسه *(الأبسط — بدون أتمتة)*

> **الإجابة:** [ ]
