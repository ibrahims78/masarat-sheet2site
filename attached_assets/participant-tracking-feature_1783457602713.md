> **الحالة:** مقترح — قيد النقاش  
> **التاريخ:** يوليو 2026  
> **النطاق:** نظام Sheet2Site — لوحة إدارة المشاريع

---

## 1. نظرة عامة على الميزة

### الهدف
تمكين مدير المشروع من تحديد قائمة بالمشاركين المطلوب منهم تعبئة النموذج، ومتابعة حالة كل منهم في الوقت الفعلي: من عبّأ ومن لم يعبّئ، مع إمكانية إرسال تذكيرات للمتأخرين.

### الفجوة الحالية
النظام الحالي يعرف **من عبّأ الفعل**، لكنه لا يعرف **من كان مفترضاً أن يعبّئ**. هذه الميزة تُغلق هذه الفجوة.

### أمثلة على حالات الاستخدام
- إدارة موارد بشرية تريد متابعة من أكمل نموذج الإجازة من 150 موظفاً
- مدرسة تريد متابعة من أعاد نموذج موافقة ولي الأمر من 200 طالب
- جهة حكومية تريد متابعة إقرارات 80 موظفاً

## 2. المقاربة المختارة: الرابط الشخصي (Personal Token Link)

### لماذا هذه المقاربة؟

| المعيار | الرابط الشخصي ✅ | مطابقة الحقل |
|---|---|---|
| دقة التتبع | 100% — لا التباس | تعتمد على كتابة صحيحة للاسم |
| التكرار | مستحيل — كل رابط يُستخدم مرة | ممكن |
| شخصان بنفس الاسم | لا مشكلة | تعارض |
| تعبئة مسبقة للبيانات | ✅ ممكنة | ❌ |
| إرسال تذكير مستهدف | ✅ بالضبط للمتأخرين | ❌ |
| التعقيد التقني | متوسط | بسيط |

### آلية العمل — خطوة بخطوة

```
1. المدير يُنشئ قائمة المشاركين في لوحة التحكم
          ↓
2. النظام يُولّد token فريد لكل مشارك
          ↓
3. كل مشارك يحصل على رابط شخصي:
   https://domain.com/r/{project-slug}/p/{participant-token}
          ↓
4. المشارك يفتح رابطه → النموذج يظهر مع اسمه مُعبّأً مسبقاً
          ↓
5. بعد التعبئة → يُربط السجل بالمشارك في قاعدة البيانات
          ↓
6. لوحة التحكم تُحدَّث فوراً: عبّأ / لم يعبّئ
```

## 3. تصميم قاعدة البيانات

### الجدول الجديد: `project_participants`

```sql
CREATE TABLE project_participants (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id    UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  name          TEXT NOT NULL,                    -- الاسم كما أدخله المدير
  identifier    TEXT,                             -- رقم جوال أو إيميل (اختياري)
  token         UUID NOT NULL UNIQUE DEFAULT gen_random_uuid(), -- الرابط الشخصي
  record_id     UUID REFERENCES project_records(id) ON DELETE SET NULL, -- ربط بالسجل بعد التعبئة
  submitted_at  TIMESTAMPTZ,                      -- وقت التعبئة
  last_notified_at TIMESTAMPTZ,                   -- آخر مرة أُرسل تذكير
  notify_count  INT NOT NULL DEFAULT 0,           -- عدد التذكيرات المُرسلة
  added_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  notes         TEXT                              -- ملاحظات المدير
);

CREATE INDEX ON project_participants(project_id);
CREATE INDEX ON project_participants(token);
CREATE INDEX ON project_participants(record_id);
```

### التعديل على `projects`

```sql
-- إضافة عمود لتفعيل الميزة وتحديد الحقل المُعبّأ مسبقاً
ALTER TABLE projects ADD COLUMN participants_enabled BOOLEAN DEFAULT false;
ALTER TABLE projects ADD COLUMN participant_name_field TEXT;  -- مفتاح الحقل الذي يُعبّأ باسم المشارك
```

## 4. نقاط API المقترحة

### إدارة المشاركين (للمدير)

| الطريقة | المسار | الوصف |
|---|---|---|
| `GET` | `/api/projects/:id/participants` | قائمة المشاركين مع حالة كل منهم |
| `POST` | `/api/projects/:id/participants` | إضافة مشارك واحد |
| `POST` | `/api/projects/:id/participants/import` | استيراد من Excel |
| `DELETE` | `/api/projects/:id/participants/:pid` | حذف مشارك |
| `POST` | `/api/projects/:id/participants/:pid/notify` | إرسال تذكير لمشارك محدد |
| `POST` | `/api/projects/:id/participants/notify-pending` | إرسال تذكير لجميع المتأخرين دفعة واحدة |

### النموذج العام (للمشارك)

| الطريقة | المسار | الوصف |
|---|---|---|
| `GET` | `/api/pform/:projectId/participant/:token` | التحقق من الرابط وإعادة بيانات المشارك |
| (النموذج يُرسل عبر الـ route الحالي) | — | عند الإرسال يُرسل `participantToken` في الـ body |

## 6. الإشعارات — الخيارات المتاحة والنقاش

### 6.1 قنوات الإشعار الممكنة

| القناة | الجاهزية | المتطلبات | الملاحظات |
|---|---|---|---|
| **رسالة واتساب** | 🟡 شبه جاهزة | رقم جوال + WhatsApp Business API | الأكثر فاعلية في السياق العربي |
| **تيليغرام** | 🟢 البنية موجودة | bot token مُعدّ في المشروع | النظام يدعم تيليغرام أصلاً |
| **إيميل** | 🟢 البنية موجودة | إعدادات SMTP في المشروع | أقل قراءةً |
| **رابط قابل للنسخ** | 🟢 فوري | لا شيء | المدير يُوزّع الرابط يدوياً |
| **QR Code** | 🟡 ممكنة | مكتبة QR | مفيد للطباعة والتوزيع وجهاً لوجه |

### 6.2 سيناريوهات الإشعار

**أ. إشعار مبدئي (عند الإضافة)**
```
خيار 1: إرسال الرابط فور إضافة المشارك تلقائياً
خيار 2: إرسال يدوي بعد مراجعة القائمة كاملة
خيار 3: المدير ينسخ الروابط ويُوزّعها بنفسه
```

**ب. التذكيرات (للمتأخرين)**
```
- يدوي: المدير يضغط "تذكير" لكل مشارك أو "تذكير الجميع"
- تلقائي بجدول: مثلاً — تذكير تلقائي قبل 48 ساعة من الموعد النهائي
- مشروط: التذكير فقط لمن مضى عليه أكثر من X أيام
```

**ج. تنسيق الرسالة**
```
مرحباً {name}،
نرجو تعبئة نموذج "{project_name}" عبر الرابط التالي:
{personal_link}
آخر موعد: {deadline}

شكراً
```

### 6.3 نقاط النقاش المفتوحة 🗣️

**1. طريقة توزيع الروابط:**
- هل يُوزّعها المدير يدوياً (يأخذ الرابط ويُرسله)؟
- أم يُرسلها النظام تلقائياً عند الإضافة؟
- أم بضغطة زر "أرسل للجميع"؟

**2. متطلبات رقم التواصل:**
- هل رقم الجوال/الإيميل إلزامي أم اختياري؟
- إذا كان اختيارياً، ما البديل لمن لا يملك تواصلاً مُدخلاً؟

**3. واتساب:**
- النظام الحالي لا يدعم إرسال واتساب مباشرةً (يحتاج WhatsApp Business API)
- بديل أبسط: إنشاء رابط `wa.me` جاهز يفتحه المدير بضغطة واحدة:
  `https://wa.me/966501234567?text=مرحباً، يُرجى تعبئة النموذج: {link}`

**4. الموعد النهائي (Deadline):**
- هل تُضاف خاصية موعد نهائي للمشروع تظهر في رسائل التذكير؟

**5. الخصوصية:**
- هل يُسمح للمشارك برؤية بيانات مشارك آخر؟ (الجواب: لا — كل رابط معزول)
- هل يرى المشارك نسبة اكتمال المجموعة؟

## 7. الجدول الزمني التقديري للتطبيق

| المرحلة | المكوّنات | التقدير |
|---|---|---|
| 1 — قاعدة البيانات | الجدول الجديد + التعديلات | يوم واحد |
| 2 — السيرفر | الـ routes (CRUD + رابط شخصي + ربط السجل) | يومان |
| 3 — واجهة المدير | تبويب المشاركين + الإحصائيات | يومان |
| 4 — تجربة المشارك | صفحة الرابط الشخصي + تعبئة مسبقة | يوم واحد |
| 5 — الإشعارات | حسب القنوات المُختارة | 1-3 أيام |
| **المجموع** | | **7–10 أيام** |
