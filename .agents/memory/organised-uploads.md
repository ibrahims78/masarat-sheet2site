name: Organised file uploads
description: How the upload path hierarchy works, key invariants for serve/delete, and backward-compat rules
---

## Rule
Files are stored at `uploads/{project-slug}/{uploadFolder}/{uuid.ext}` instead of the old flat `uploads/{uuid.ext}`.

- `slugify(name)` converts Arabic/English project names to safe ASCII slugs (max 50 chars, falls back to "project").
- `uploadFolder` is a UUID generated by the client once per form session; sanitised server-side to `[a-zA-Z0-9_-]` max 64 chars.
- `organizeUploadedFile()` in `server/middleware/upload.ts` is called **after** all validation (validateMimeType + validateFieldRestrictions) because both validators read the file from the flat uploadsDir before the move.

## Serving (`server/index.ts`)
Route is `/uploads/*` (wildcard, not `:filename`).
Path traversal guard: `path.normalize(rawPath)` → `path.join(uploadsDir, normalized)` → must start with `uploadsDir + path.sep`.
IDOR check uses `path.basename(normalized)` (the UUID leaf filename) which is globally unique — still works for both flat and nested.

## Deleting
**Always** use `val.slice('/uploads/'.length)` to get the relative path, then `path.normalize()` + traversal guard before `fs.unlink`.
Never use `path.basename(val)` for deletion — it silently fails for nested paths.
All four deletion sites in `server/routes/projects.ts` now use the safe relative-path pattern.

## Backward compat
Requests that don't send `uploadFolder` fall back to the old flat path automatically (the `if (uploadFolder)` guard in `buildResponse`).

**Why:** Flat paths break `fs.unlink` and the wildcard serve route needed to be `/uploads/*` instead of `/uploads/:filename` to handle nested paths. The old basename-only approach silently left organised files on disk when records were deleted.
